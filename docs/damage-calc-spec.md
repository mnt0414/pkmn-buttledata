# ダメージ計算機能 技術選定・要件定義

対象: pkmn-buttledata（ダブルバトル記録アプリ）への「ポケモンチャンピオンズ ダブル」向けダメージ計算機能の追加。
作成日: 2026-06-23 / ステータス: 要件定義（デザインは後日 Claude Design で確定）

---

## 1. ゴール

- チャンピオンズ ダブル基準のダメージ計算機をアプリ内に追加する。
- 自分の登録パーティのポケモン（育成内容込み）を計算機にワンタップで呼び出せる。
- パーティ登録時にチャンピオンズ固有の「能力ポイント（努力値）」配分を登録できる（任意機能）。
- 既存の「単一HTML＋CSV・ビルドなし・GitHub Pages・localStorage」構成を壊さない。
- 後日のデザイン差し替え（Claude Design）に耐える構造にする。

---

## 2. 既存アプリの現状（前提）

| 項目 | 内容 |
|---|---|
| 構成 | `index.html` 単一ファイル（CSS/JS埋め込み）＋ `pokemon_list.csv` |
| フレームワーク | なし（vanilla JS）。ビルド工程なし |
| データ | CSV 1119件。列は `no,name,type1,type2,sprite_override,sprite_url,yakkun_url` のみ |
| 状態保存 | localStorage キー `pb3` = `{parties:[{id,name,pokemon:[名前]}], activeParty, battles:[]}` |
| ポケモン識別子 | 日本語名（CSVの `name`）。図鑑番号 `no` も保持 |
| 画面 | パーティ / 対戦記録 / データ / 設定 の4タブ |
| 同期 | Google Apps Script(GAS) によるクラウド同期、JSON入出力 |

致命的なギャップ:
CSVに「種族値・とくせい・技・タイプ相性」が一切ない。ダメージ計算には全て必須。

## 3. 技術選定

### 3.1 計算エンジン: `@smogon/calc` を採用（確定）

採用理由:
- 種族値・技・特性・道具・タイプ相性・第9世代テラスタルまで全データを内蔵 → CSV拡張で全データを自前調達する必要がなくなる。
- 計算式・乱数16通り・急所・天候/フィールド/ダブル拡散(0.75倍)・確定数を正確に処理。Pokémon Showdown と同等エンジン。
- CDN(UMD)で `<script>` 2本のみ、ビルド不要。既存方針を維持できる。

```html
<script src="https://cdn.jsdelivr.net/npm/@smogon/calc@0.11/dist/data/production.min.js"></script>
<script src="https://cdn.jsdelivr.net/npm/@smogon/calc@0.11"></script>
```
バージョンはピン留め（@0.11）。グローバルに `calc`（`calculate/Pokemon/Move/Generations`）が生える。

重要な前提（調査で判明）:
`@smogon/calc` のデータはSV(第9世代)まで。チャンピオンズはSVから多数の差分があり、エンジン単体では正確に計算できない。そのため §3.5 の「Champions差分オーバーレイ」を必須コンポーネントとして併用する。

### 3.2 採用方針の比較

| 観点 | @smogon/calc（採用） | 自作 vanilla JS |
|---|---|---|
| データ調達 | 内蔵（不要） | 種族値/技/特性/相性を全件自前調達（膨大） |
| 計算正確性 | 実戦検証済み | テラ/特性/道具の例外を自前実装、バグ温床 |
| 工数 | 連携と日本語化のみ | 非常に大 |
| 弱点 | 英語名のため日英対応辞書が必要／CDN依存 | 全て日本語で完結 |

### 3.3 日英対応辞書（採用に伴う必須準備物）

`@smogon/calc` は英語名で識別するため、UI（日本語）↔エンジン（英語）の変換辞書を静的JSONで用意する。

| 辞書 | キー→値 | 調達元 |
|---|---|---|
| ポケモン | 図鑑番号/日本語名 → 英語種族名 | PokéAPI(pokemon-species.names) で一括生成 |
| 技 | 日本語技名 → 英語技名 | PokéAPI(move.names) |
| とくせい | 日本語 → 英語 | PokéAPI(ability.names) |
| もちもの | 日本語 → 英語 | PokéAPI(item.names) |
| 性格 | 日本語 → 英語 | 固定25種、手書き |
| タイプ/テラス | 日本語 → 英語 | 固定18種、手書き |

- 辞書は事前に1回スクリプトで生成し `data/dict_ja_en.json` として同梱（実行時はfetchするだけ）。
- 既存CSVは図鑑番号 `no` を持つので、ポケモンは番号で確実に英語名へ引ける。

### 3.5 Champions差分オーバーレイ（必須・調査で確定）

出典: yakkun.com/ch/ 各ページ（changes / move_changes / ability_list?new=1 / zukan/new_mega）。
SV→チャンピオンズの差分のうち「ダメージ計算に影響するもの」を静的データ＋手当てで補正する。

ダメージに影響する差分（対応必要）:
- 威力/命中の変更技。例: ボーンラッシュ25→30、であいがしら90→100、かげぬい80→90、トロピカルキック70→85、くちばしキャノン100→120、ほのおのムチ80→90、Gのちから80→90、りんごさん80→90、ナイトバースト85→90、ひゃっきやこう60→65、バリアーラッシュ70→90、ひょうざんおろし100→120、クラブハンマー命中90→95 ほか。→ `Move` の威力を override して注入。
- 新メガシンカ（ZA由来。例: メガライチュウX/Y）。@smogon/calc に種族未収録の可能性大。→ 種族値・タイプ・特性を yakkun から収集し、Pokemon の overrides（baseStats/types/ability）＋自前算出の実数ステータスで注入。
- 新特性のうち威力倍率系。例: ほのおのたてがみ(炎技1.5倍)、ドラゴンスキン(ノーマル→ドラゴン化＋1.2倍)、メガソーラー(自分の技時のみ晴れ扱い)。→ エンジンが知らない特性は、既存同等特性へのマッピング（例: 既知の似た特性名を渡す）か、計算後に倍率を手動適用する。

ダメージ計算に影響しない差分（対応不要 or UXのみ）:
- 個体値撤廃(全31固定)・レベル50統一 → §4 の実数ステータス注入で既に吸収。
- PPの4段階統一(8/12/16/20)・ポイントアップ廃止 → 計算に無関係。
- 覚える技の変更(没収/追加) → 技候補リストのUXに反映するのみ（任意）。
- まもる弱体化・状態異常仕様変更・ゼンブイリングの運用 → 単発ダメージ数値には不要。

オーバーレイのデータ構造（同梱JSON案）:
```js
// data/champions_overlay.json
{
  "moves":    { "くちばしキャノン": { "basePower": 120 }, ... },
  "abilities":{ "ほのおのたてがみ": { "type":"ほのお", "mult":1.5 },
                "ドラゴンスキン":   { "fromType":"ノーマル","toType":"ドラゴン","mult":1.2 }, ... },
  "species":  { "メガライチュウY": { "baseStats":{...},"types":["でんき"],"ability":"..." }, ... }
}
```
- レギュレーション更新（例: レギュMB 2026/6/17 で新ポケ追加）で差分が増えるため、オーバーレイは更新前提の独立ファイルにする。

収集状況（2026-06-24 完了 / `data/champions_overlay.json`）:
- moves 21件（威力/命中/効果変更）、abilities 6件（うち倍率系3つ＝ほのおのたてがみ1.5倍/ドラゴンスキン1.2倍/メガソーラーは構造化済）。
- megas 98件（種族値・タイプ・特性）。うち `data-gen=9` のZA新メガが46件で、これらは @smogon/calc(第9世代)に非収録 → overlayで種族注入が必須。旧メガ(XY/ORAS)52件も第9世代エンジンには存在しないため同様に注入対象。
- 注意: ZA新メガ11件（メガライチュウX/Y除く一部、メガアブソルZ・メガセグレイブ等）は yakkun側で特性が未掲載（`tokusei=0`）。`ability_pending:true` を付与。種族値・タイプは確定だが、特性依存の補正はyakkun掲載後に追補する。

### 3.4 ホスティング・構成への影響

- 追加ファイル: `<script>` 2本（CDN）、`data/dict_ja_en.json`（静的）、必要に応じ `data/moves_by_pokemon.json`（各ポケモンの覚える技候補。任意・UX向上用）。
- ビルド工程は引き続き不要。GitHub Pages のまま。
- オフライン: CDN依存のため初回はオンライン必須。将来 `@smogon/calc` のdistを同梱すれば完全オフライン化可能（任意）。

## 4. チャンピオンズ努力値仕様と連携方式

出典: https://yakkun.com/ch/stat_points.htm

### 4.1 能力ポイント（Stat Points）ルール

| 項目 | 値 |
|---|---|
| 1能力あたり上限 | 32 |
| 合計上限 | 66 |
| 個体値 | 31固定 |
| レベル | 50固定 |
| 性格補正 | 0.9 / 1.0 / 1.1 |
| デフォルト（ぶっぱ） | 2能力に32、1能力に2（計66） |

能力値計算式（端数切り捨て）:
- HP = (種族値×2 + 31 + ポイント×2) × 50 / 100 + 60
- HP以外 = ⌊{(種族値×2 + 31 + ポイント×2) × 50 / 100 + 5} × 性格補正⌋

### 4.2 @smogon/calc への連携方式（重要）

`@smogon/calc` の入力は SV式の努力値(0〜252・合計510・4刻み)で、チャンピオンズの(0〜32・合計66・+1刻み)を努力値換算すると最上限で+1ズレる。
→ 努力値入力は使わず、上式で「最終6ステータス」をアプリ側で算出し、`@smogon/calc` の Pokemon に実数ステータスとして注入する（個体値31・Lv50・性格補正込み）。

- 注入APIの具体手段（Pokemonのstats上書き or overrides）は実装フェーズで `@smogon/calc@0.11` の仕様を実機確認して確定する（未検証リスクとして §10 に記載）。

## 5. 機能要件

### F1. ダメージ計算機（新タブ「計算」）

- 攻撃側・防御側の2枠。各枠で設定:
  - ポケモン選択（既存の検索モーダルを再利用）
  - 能力ポイント配分（§4.1。または最終ステータス直接確認）
  - 性格、とくせい、もちもの、テラスタル(タイプ/有無)、ランク補正(各能力 -6〜+6)
- 技選択（複数技まとめて計算 = 参考サイト準拠）。
- 場の条件: 天候、フィールド、ダブル時の拡散補正(0.75)、リフレクター/ひかりのかべ 等。
- 出力:
  - ダメージ範囲（最小〜最大）
  - 残HPに対する割合（%）
  - 乱数/確定数（例: 確定2発、乱数1発(75%) など）
- 参考サイトの「双方向同時計算」は将来拡張（まず片方向で確定）。

### F2. 自パーティ呼び出し

- 計算機の攻撃/防御枠に「自分のパーティから選ぶ」導線を置く。
- 選ぶとそのポケモンの育成内容（能力ポイント/性格/特性/持ち物/テラス/技）が枠に流し込まれる。
- 育成内容が未登録のポケモンはデフォルト（ぶっぱ・無補正・テラなし）で読み込む。

### F3. パーティ育成データ拡張（任意機能）

- パーティ登録時、各ポケモンにチャンピオンズ能力ポイントを設定可能にする。
- UI: 6能力スライダー/数値入力。上限32・合計66をリアルタイム検証。プリセット「ぶっぱ(32/32/2)」を1タップ適用。
- あわせて 性格・とくせい・もちもの・テラスタイプ・技(最大4) も登録可能に。
- 任意機能なので、未設定でも既存の記録フローは従来通り動く。

### F4. データ移行（既存ユーザー保護）

- 既存 `pb3.parties[].pokemon` は「日本語名の文字列配列」。これをオブジェクトに拡張:
```js
// 旧: pokemon: ["ガブリアス", ...]
// 新: pokemon: [{ name:"ガブリアス", points:{H:0,A:32,B:0,C:0,D:2,S:32},
//                nature:"いじっぱり", ability:"さめはだ", item:"いのちのたま",
//                tera:"じめん", moves:["じしん","げきりん","",""] }, ...]
```
- 起動時に文字列要素を検出したら新形式へ自動マイグレーション（name以外はデフォルト）。
- 旧バージョンとの相互運用: 表示系は `p.name || p`（文字列/オブジェクト両対応）で参照。
- GAS同期・JSON入出力のスキーマも新形式に追従（後方互換の読み込みを担保）。

## 6. 非機能要件

- パフォーマンス: 辞書JSONは初回fetch後メモリ保持。計算は同期で即時。
- 互換性: モバイルファースト（既存CSS踏襲）。最大幅900px。
- 信頼性: CDN/辞書のロード失敗時はバナー表示（既存 `csv-error` パターン流用）。
- セキュリティ: 外部送信なし（計算は全てクライアント内）。CDNはSRI付与を検討。

## 7. デザイン差し替え対応方針（Claude Design 連携）

後日デザインを別途作る前提で、ロジックと見た目を分離しておく。

- ロジック層: `calcEngine`（入力→@smogon/calc→結果オブジェクト）を純関数群として独立させ、DOMに依存させない。
- ビュー層: `renderCalc()` がデータを受け取り描画。デザイン変更時はこことCSSのみ差し替え。
- マークアップ: 安定した id / `data-*` 属性（例 `data-role="atk-poke"`）を付け、Claude Design 側がフックしやすくする。
- テーマ: 既存の CSS 変数（`--bg0/--accent` 等）をデザイントークンとして継続利用。
- 計算結果のデータ構造を先に固定 → デザインは後から自由に乗せ替え可能。

## 12. フェーズ0 検証結果（2026-06-24, @smogon/calc@0.11）

Node＋ブラウザ(Playwright)の両方で実機検証。スパイクは scratchpad（リポジトリ非同梱）。

### 12.1 CDN読込方式（確定）
- 採用: `<script type="module">import * as calc from 'https://cdn.jsdelivr.net/npm/@smogon/calc@0.11/+esm'</script>`。`Pokemon/Move/Field/calculate/Generations` 等を全取得、ビルド不要。
- 不可と判明: 素のmin2本構成（`dist/data/production.min.js` ＋ `dist/production.min.js`）。後者に `require("./desc")` が残存しブラウザで `require is not defined` で落ちる。bare URL `.../@smogon/calc@0.11` は CommonJS `dist/index.js` に解決されるため同様に不可。
- 既存アプリは非module inline scriptのため、module側で `window.calc = calc` に橋渡しし、計算コードは読込完了(await)後に呼ぶ。
- 可用性/再現性を上げるなら `/+esm` の出力を `data/smogon-calc.esm.js` にvendoringして相対importに切替（任意）。

### 12.2 gen9ダブル・スプレッド（確定）
- `new Field({gameType:'Doubles'})` で範囲技が自動0.75倍。検証: じしんがシングル97-115→ダブル73-87（比0.757）。

### 12.3 実数ステータス注入 R1（解消・最重要）
- `calculate()` は内部で `attacker.clone()` する。`clone()` は `ivs/evs/nature/overrides(species)` から再構築するため、構築後に `pokemon.stats` を手代入しても無視される（要注意）。
- 注入経路: 中立性格(`nature:'Hardy'`)＋`ivs` 全31＋`evs` 全0に固定し、`overrides.baseStats` を目標実数から逆算。
  - Lv50・IV31・EV0・中立では `非HP実数 = baseStats+20`、`HP実数 = baseStats+75`。
  - よって `baseStats[stat] = 目標実数 - 20`（HPのみ `- 20→-75`）。性格補正はアプリ側のチャンピオンズ計算で実数に織り込み済みのため、エンジン側は中立で渡す。
- 検証: 注入Garchomp と 実Garchomp(意地252)の `.stats` 完全一致、じしんダメージも 97-115 で完全一致（clone後も保持）。
- 補足: `baseStats` 逆算値が負でも `calcStat` はクランプせず素の式で算出するため、極端な実数（例: ヌケニンHP1）でも成立。

### 12.4 メガ注入（オーバーレイ megas 98件に直結）
- 第9世代エンジンにメガは非収録だが、任意のbase種に対し `overrides:{ baseStats(注入), types:[...] }` ＋ `ability` を与えれば種族値・タイプ・特性を差し替え可能。検証: Pikachuベースに ZA新メガ想定の種族値/電・ドラゴン/特性を注入し計算成立。
- 実装では「メガ前のポケモンをbaseに、overlayの megas[名前] から baseStats/types/abilities を適用」。`ability_pending:true` の11件は特性空のまま種族値・タイプのみ適用。

### 12.5 技威力など上書き（オーバーレイ moves 21件）
- `new Move(gen, name, { overrides: { basePower: 120 } })` で威力差し替え→ダメージ変化を確認（くちばしキャノン 100→120 で 77→92）。命中・タイプ等も `overrides` で同様に上書き可能。

### 12.6 新特性の倍率（R6・未スパイク／方針）
- ほのおのたてがみ(炎技1.5×)・ドラゴンスキン(ノーマル→ドラゴン1.2×)・メガソーラー(自分の技時のみ晴れ扱い) はエンジン非対応。`champions_overlay.json.abilities` の倍率を用い、計算後にダメージ範囲へアプリ側で乗算するラッパーで対応（フェーズ2で実装）。

## 13. フェーズ1 検証結果（2026-06-24, 日英辞書）

### 13.1 成果物（`data/` に同梱）
PokeAPIのCSV（github master の data/v2/csv）をディスクへ取得しNodeで結合・エンジン照合して生成。各JSONは `{ _meta, map:{ 日本語: エンジン表示名 } }` 形式。
- `dict_pokemon.json` 1092件 / `dict_moves.json` 869件 / `dict_abilities.json` 305件 / `dict_items.json` 484件 / `dict_natures.json` 25件。

### 13.2 重要な実装上の確定事項
- 辞書の値は __toIDキーではなくエンジンの表示名__ で持つ（例 `いのちのたま → "Life Orb"`、`ガブリアス → "Garchomp"`、`ロトム(ウォッシュロトム) → "Rotom-Wash"`）。理由: `@smogon/calc` の `Pokemon` はアイテム/特性を __toID正規化せず表示名で一致__ するため、`"lifeorb"` を渡すと Life Orb が無効化される（実測 97-115、表示名 `"Life Orb"` なら 126-149）。species/move は `get()` 側がtoIDするのでキーでも通るが、全dictを表示名に統一。
- ポケモンの写像は __日本語名を正本__ とする（アプリ `pokemon_list.csv` の `no`/sprite 列は名前とズレている箇所が多いため、`no` 基準は使わない）。
- フォルムは「和語フォルム接尾辞 → Showdown接尾辞」の対応表で解決（アローラ/ガラル/ヒスイ/パルデア=地域、れいじゅう=therian、ロトム家電・キュレム黒白・オーガポンの面・パルデアケンタロス三種・バドレックス黒馬白馬・ネクロズマ各形態など個別キーは明示マップ）。`ギルガルド→Aegislash-Shield`、`エナモラス(=ラブトロス表記揺れ)→Enamorus` はALIASで吸収。
- E2E実計算で検証（カビゴン受け）: ガブリアス+じしん+いのちのたま=126-149、ランドロス(れいじゅう)+じしん+こだわりハチマキ=142-168、テツノカイナ+インファイト+ブーストエナジー(クォークチャージ)=246-290。名前→辞書→計算が一貫して成立。

### 13.3 アプリ `pokemon_list.csv` の既存データ不整合（本機能とは独立・要修正候補）
辞書生成の副産物として検出。ダメージ計算自体は日本語名正本で正しく動くが、アプリのスプライト表示等に影響する可能性あり。
- 実在しない/誤記の和名 9件（マップ不能）: モコシャン(no=180), キノコ(283), ドスコイ(305), ロゼイド(406), コウモリ(527), ジュフォレスト(596), アイスキャッスル(617), フタチキ(680), ブルガネット(709)。
- `name` と `no`/sprite列がズレている行 32件（例 no=276 の行名が「キャモメ」だが no=276 は本来スバメ）。主に 213, 276〜289, 304, 407〜412, 528, 595, 616, 618, 690〜691, 708, 712, 734〜735 付近。
- `タイプ:ヌル`(no=772) は半角コロン。PokeAPI正名は全角「タイプ：ヌル」（辞書側は正規化して対応済）。

辞書再生成スクリプトは scratchpad（`build_dict.js`、リポジトリ非同梱）。レギュ更新時はPokeAPI CSV再取得＋再実行で更新可。

## 14. フェーズ2 検証結果（2026-06-24, 計算UI・片方向）

`index.html` 単一ファイルに計算タブを追加（ビルドなし維持）。ロジック層 `calcEngine`（DOM非依存の純関数群）とビュー層 `recalc/renderCalc` を分離。Node＋ブラウザ(Playwright)実機で検証。

### 14.1 構成・読込
- ESMブリッジ: 末尾に `<script type="module">import * as smogon from '.../@smogon/calc@0.11/+esm'; window.SMOGON=smogon; dispatchEvent('smogon-ready')</script>` を追加。非module側の計算コードは `whenSmogon()` で `smogon-ready` を待ってから初期化。
- 辞書/overlayは計算タブ初回表示時（`openCalc`）に `./data/*.json` を遅延fetch（CSVローダと同じ cache-bust）。`CALC.ready` 後に `recalc()`。
- 計算タブを開くまでエンジン/辞書を読まない＝既存タブの初期表示は従来通り軽量。

### 14.2 calcEngine（純ロジック・主要関数）
- `computeChampStats(base,pts,natureJa)`: §4.1の式で6実数を算出（性格補正は `gen.natures` の plus/minus から ±10%）。HP種族値1（ヌケニン）は1固定。
- `injBase(final)`: R1注入。非HP=実数-20／HP=実数-75。`buildSide` が 中立Hardy＋IV31＋EV0＋`overrides.baseStats` でPokémon生成。検証: 注入後の `pokemon.stats` が `computeChampStats` 出力と全6完全一致（ガブリアス/カビゴン/テツノカイナ/ヌケニンHP1）。
- `prepMove(moveJa,atk)`: overlay `moves[ja]` の `basePower/accuracy` を `Move.overrides` で上書き。新特性のタイプ変換（ドラゴンスキン: ノーマル→ドラゴン）は型を上書きしSTAB/相性をエンジンに正しく計算させ、倍率はpost-mult。
- `abilityMult`(prepMove内): ほのおのたてがみ(炎技×1.5)／ドラゴンスキン(×1.2)／メガソーラー(自分の技時=天候はれ扱い→`buildField`で晴れ付与)。エンジン非対応の新特性は `ability:undefined` で渡し、倍率は計算後にダメージ範囲へ乗算。
- `buildField(f,atkAbility)`: gameType(ダブル既定で範囲技0.75自動)、天候/フィールド、`defenderSide`=リフレクター/ひかりのかべ、`attackerSide`=てだすけ、`move.isCrit`=急所。
- `calcOnce` 出力: min/maxダメージ、防御側HP、%、確定/乱数（`simpleKO`、JA統一）、overlay適用ノート。`res.desc()`(英語)は注入の都合でEV/性格ラベルが「0 Atk」等と誤表示になるためUI非表示（数値は注入実数で正しい）。

### 14.3 実機検証（合格）
- 注入往復一致: 上記4種で `pokemon.stats===computeChampStats`。
- ダブルスプレッド: じしん single 88-105 → double 66-78（比0.743）。
- overlay技威力: であいがしら 威力90→100 でダメージ上昇（54-64→60-71）。
- 新特性×1.5: かえんほうしゃ＋ほのおのたてがみ で max が ×1.5（51-61→76-91）。
- メガ注入: メガリザードンY（種族値/タイプ/特性差替）で計算成立。
- もちもの表示名経路: こだわりハチマキ ×1.5（78→118）、いのちのたま反映（`res.desc()`に "Life Orb Garchomp"）。
- テラスタル: テラスじめんでじしんのSTAB上昇（1.5→2.0、78→104）。
- 壁: ダブルのリフレクターは ×2/3（0.667）でエンジン適用（シングルの0.5ではない＝正しい）。
- ブラウザE2E: ESM読込・辞書fetch(pokemon1092/move869/mega98)・datalist(特性311/技870)・性格25・UI入力→委譲recalc→結果描画まで一気通貫。例: ガブリアス(意地AS252相当/いのちのたま)→カビゴン(HB)で じしん86-101(乱数3〜4)・げきりん138-164(確定2)。

### 14.4 既知の簡略化（フェーズ5以降の候補）
- 確定数は `simpleKO`（range/HPの単純割り）。残飯・砂・反動などの残留ダメージは未考慮（overlay倍率との一貫性優先）。エンジン `kochance()` 併用は後日。
- 防御側のチャンピオンズ新特性（防御倍率系）は未対応（SV既知特性は名前一致で有効）。
- かんつうドリル（まもり貫通1/4）等、倍率以外の新特性効果は未対応（ノート表示のみ将来）。
- 双方向同時計算は未実装（フェーズ5）。自パーティ呼び出しはフェーズ4。

## 15. フェーズ3 検証結果（2026-06-24, 育成データ）

`index.html` 単一ファイルに育成データ機能を追加（ビルドなし維持）。パーティの各ポケモンに能力ポイント等を保存し、フェーズ4で計算機へ流し込む土台。

### 15.1 データスキーマ（F4 / 確定）
- 旧: `parties[].pokemon = ["ガブリアス", ...]`（日本語名の文字列配列）。
- 新: `parties[].pokemon = [{ name, points:{H,A,B,C,D,S}, nature, ability, item, tera, moves:[4] }, ...]`。
- 自動移行 `migrateParties()`: `pokemon` 配列内の文字列要素を `{name}` に変換し `save()`。起動時(INIT)・GAS取得後(`pullGAS`)に実行。文字列が無ければ無変換。
- 後方互換参照: `pname(x)`=名前取り出し（文字列/オブジェクト両対応）、`pbuild(x)`=オブジェクト化。描画系（パーティ/対戦/選出グリッド）は全て `pname()` 経由に変更。
- GAS/JSON: 同期はS全体をJSONで往復するためスキーマ追従は自動。`exportData` は新形式をそのまま出力。`pullGAS` は取得後に `migrateParties()` で旧形式も吸収。`importData` は従来通り battles のみ取り込み（パーティ育成はGAS pull / 全体エクスポートで移送）。

### 15.2 UI（F3）
- パーティ編集スロットをタップ → 育成データモーダル（`#build-modal`、既存 `.modal` 流用）。空きスロットは従来通りポケモン選択モーダル。
- 入力: 能力ポイント H/A/B/C/D/S（数値・各0〜32）＋「ぶっぱ」プリセット（2/32/…/32=計66）＋合計表示（66超で赤）、性格(25)、テラスタル(18+なし)、とくせい/もちもの/技×4（datalist `dl-ability/dl-item/dl-move` を共有）。
- 「ポケモンを変更」: 育成モーダルを閉じてから選択モーダルを開く（同z-index重なり回避）。種族変更時は育成内容をリセット（別ポケの特性/技は無効なため）。
- 育成データ有りのスロットに緑ドット表示（`hasBuild()`：ポイント>0・特性・道具・テラ・技・非まじめ性格のいずれか）。
- 辞書は計算タブと共有。`loadCalcDicts()` をエンジン(ESM)待ちから分離し、育成エディタ初回オープン時にJSONのみ遅延fetch（エンジン未ロードでも編集可）。

### 15.3 実機検証（合格 / Playwright）
- 移行: 旧形式 `["ガブリアス","カビゴン","テツノカイナ"]` を投入しリロード → 3件とも `{name}` 化・localStorage永続化・スロット名表示。
